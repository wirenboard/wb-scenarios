# PID Controller — Архитектурная документация (ARC42)

## 1. Введение и цели

### 1.1. Описание задачи
Сценарий **pidController** реализует ПИД-регулятор для управления аналоговыми выходами.
Сценарий записывает вычисленные значения напрямую в аналоговые выходы.

### 1.2. Функциональные требования
- Пересчёт ПИД-алгоритма с настраиваемым интервалом
- Маппинг выхода ПИД (0-100%) в единицы актуатора (min..max)
- Поддержка прямого и инвертированного режима актуаторов
- Управление массивом актуаторов с индивидуальными диапазонами
- Сохранение уставки в persistent storage между перезапусками
- Обработка ошибок датчика и актуаторов с debounce-таймером
- Автоматическое отключение при критических ошибках
- Виртуальное устройство для управления через WebUI

### 1.3. Качественные цели
| Приоритет | Цель | Описание |
|---|---|---|
| 1 | Надёжность | Корректная работа при ошибках датчиков/актуаторов |
| 2 | Безопасность | Сброс актуаторов в безопасное состояние при ошибках |
| 3 | Простота | Минимум настроек для типового случая |
| 4 | Гибкость | Поддержка разных типов аналоговых выходов |

### 1.4. Стейкхолдеры
| Роль | Ожидания |
|---|---|
| Пользователь контроллера | Настройка через WebUI, стабильное регулирование |
| Разработчик wb-scenarios | Соблюдение паттернов ScenarioBase |
| Инсталлятор | Минимальная конфигурация при монтаже |

## 4. Стратегия решения

### 4.1. Архитектура
Наследование от `ScenarioBase` с реализацией трёх обязательных методов:
- `generateNames()` — генерация имён VD и правил
- `validateCfg()` — валидация конфигурации
- `initSpecific()` — создание контролов, правил, запуск цикла

### 4.2. Цикл расчёта (без ШИМ)
Каждые `calculationPeriodSec` секунд:
1. Чтение датчика и уставки
2. Вычисление ПИД-выхода (0-100%)
3. Маппинг в единицы актуатора (direct/inverted)
4. Запись в MQTT (только при изменении)

### 4.3. Маппинг выхода
```
direct:   value = outputMin + (pidOutput / 100) * (outputMax - outputMin)
inverted: value = outputMax - (pidOutput / 100) * (outputMax - outputMin)
```

Каждый актуатор имеет собственные `outputMin`/`outputMax`.

```

├── ctx:
│   ├── pid              (PidEngine instance)

│     ruleSetSetpoint, rulePidReset, ruleSensorErr

│   → {controls: [sensor, ...actuators]}
│
├── validateCfg(cfg)
│   → проверки: limits, coefficients, period,
│     deadband, actuators (min < max)
│
└── initSpecific(name, cfg)
    1. Restore setpoint from storage
    2. Add VD controls
    3. Create rules
    4. Create PidEngine
    5. Start calculation cycle
```

### 5.3. Правила (rules)

| Правило | Триггер | Действие |
|---------|---------|----------|
| `sensor_changed` | Датчик | Обновить VD current_value |
| `set_sc_status` | rule_enabled | Start/stop цикл, reset PID |
| `set_setpoint` | setpoint | Сохранить в storage |
| `pid_reset` | Кнопка | Reset интеграла, restart |
| `sensor_error` | sensor#error | Debounce then disable |
| `actuator_err_N` | actuator#error | Debounce then disable |

## 7. Обработка ошибок

### 7.1. Критические ошибки (r/w)
1. Получение ошибки — запуск debounce-таймера (10 сек)
2. Если ошибка сохраняется — `USED_CONTROL_ERROR`, rule_enabled = readonly + false
3. Все актуаторы сбрасываются в безопасное состояние

### 7.2. Очистка ошибки
1. Ошибка пропала — отмена debounce-таймера
2. Проверка всех источников (датчик + все актуаторы)
3. Если все чисто — снятие readonly, state = NORMAL

## 8. Persistent Storage

Хранимые данные:
- `setpoint` — уставка, заданная пользователем
- `rule_enabled` — состояние включения (через базовый класс)

Ключ хранения: `scenariosRegistry[idPrefix].userSettings.setpoint`

При восстановлении:
1. Загрузка из storage
2. Проверка что значение в пределах [min, max]
3. Если невалидно — сброс к значению из конфига

## 9. Технический долг и перспективы

### 9.1. Output Ramp (ограничение скорости выхода)
**Статус**: Не реализовано, запланировано.

Ограничение максимальной скорости изменения выхода (units/cycle) для защиты
актуаторов (клапаны, приводы) от резких скачков.

**Проблема**: При малом значении ramp интеграл продолжает копиться, но
реальный выход меняется медленно. Это создаёт эффективный windup —
после снятия ramp-ограничения выход прыгает. Требуется реализация
back-calculation anti-windup или отслеживание реального выхода в интеграторе.

### 9.2. Заморозка интеграла в мёртвой зоне
**Статус**: Не реализовано, под наблюдением.

Текущее поведение: gain factor 0.1 (интеграл копится медленно).
Альтернатива: полная заморозка (интеграл не копится).

Для аналоговых выходов медленный дрейф может вызывать ненужное движение
актуаторов. Если на практике это станет проблемой — добавить опцию
`freezeIntegralInDeadband` в PidEngine.

### 9.3. Сохранение интеграла
**Статус**: Не реализовано, низкий приоритет.

При перезапуске wb-rules интеграл сбрасывается в 0. Система плавно
выходит на режим за несколько циклов. Для критичных процессов можно
добавить сохранение интеграла в persistent storage.

### 9.4. Авто-настройка (auto-tuning)
**Статус**: Не реализовано, низкий приоритет.

Метод Ziegler-Nichols (relay method) для автоматического подбора
коэффициентов Kp, Ki, Kd.
