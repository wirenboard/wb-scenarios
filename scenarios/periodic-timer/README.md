# Сценарий периодического таймера `periodicTimer`

## Общее описание

Данный сценарий позволяет автоматически выполнять набор действий с заданным интервалом в минутах в пределах настраиваемого временного окна. Дополнительно через указанную паузу может выполнить второй набор действий. Идеально подходит для:

- Периодического полива или орошения (включить насос → подождать → выключить)
- Цикличной вентиляции помещений
- Импульсного управления нагрузкой (например, включить на 5 минут каждый час)
- Напоминающей автоматики (звонок, сигнал и т.д.)

**О временном окне:** Сценарий срабатывает только внутри заданного временного окна (поля «Активен с» и «Активен до»). По умолчанию окно охватывает весь день (00:00–23:59). Поддерживается перенос через полночь: например, 22:00–06:00. Граница «Активен до» не входит в окно — сценарий останавливается до наступления этого времени.

**Об интервале:** Интервал отсчитывается от начала временного окна (`activeFrom`). При интервале 60 минут и `activeFrom = 08:15` сценарий срабатывает в 08:15, 09:15, 10:15 и т.д.

## Логика работы

### Автоматическое выполнение (каждую минуту)

Каждую минуту сценарий определяет текущую фазу цикла:

```
phase = минутаСутки % интервал

Если пауза > 0:
  phase < пауза  → выполнить startControls (фаза запуска)
  phase >= пауза → выполнить stopControls  (фаза остановки)

Если пауза = 0:
  phase === 0    → выполнить startControls (один раз за интервал)
```

Это означает:
- В течение первых `пауза` минут каждого интервала — startControls выполняются каждую минуту
- В оставшиеся минуты интервала — stopControls выполняются каждую минуту

> **Важно:** Действия выполняются каждую минуту повторно. Поэтому допустимы только идемпотентные типы действий: «Включить», «Выключить», «Установить значение». Типы «Переключить», «Увеличить значение», «Уменьшить значение» не поддерживаются.

### Очистка при выходе из окна

Если сценарий находился внутри окна, а затем время вышло за его пределы — автоматически выполняется `stopControls` (один раз). Это гарантирует, что контролы не останутся «включёнными» после окончания окна.

> **Ограничение:** Очистка срабатывает только в рамках текущей сессии wb-rules. При перезапуске wb-rules снаружи окна очистка не выполнится.

### Как задать activeTo

`activeTo` — это **эксклюзивная** граница: минута `activeTo` уже вне окна. Удобный способ думать об этом:

> `activeTo` = первая минута, когда ничего не должно происходить

Чтобы последний запуск `startControls` был в момент T и отработала пауза:

> `activeTo = T + пауза + 1 мин`

Пример: интервал=60 мин, пауза=5 мин, желаемый последний запуск в 19:00:
- `startControls` в 19:00–19:04, `stopControls` в 19:05
- Задать `activeTo = 19:06`

### Ручной запуск

- Кнопка «Выполнить сейчас» немедленно выполняет `startControls`
- Если `пауза > 0` — через `пауза` минут выполнятся `stopControls`
- Не влияет на автоматическое расписание
- Не переживает перезапуск wb-rules (таймер остановки хранится только в памяти)

### Состояние (`state`)

Контрол «Состояние» в виртуальном устройстве отображает текущий статус сценария:
- **Активен** — сценарий включён, сегодня рабочий день, текущее время в окне → каждую минуту применяется нужная фаза
- **Ожидает** — сценарий включён, но одно из условий не выполнено (не рабочий день или вне окна)
- **Отключен** — сценарий выключен (`rule_enabled = false`)

## Параметры конфигурации

### Активен с / Активен до (`activeFrom` / `activeTo`)

Временное окно активности сценария в формате `ЧЧ:ММ`:

- **activeFrom** (string): Начало окна (включительно). По умолчанию `00:00`
- **activeTo** (string): Конец окна (не включается). По умолчанию `23:59`

Поддерживается перенос через полночь: если `activeFrom > activeTo` (например, `22:00` — `06:00`), сценарий активен с 22:00 до 05:59 включительно.

`activeTo` не входит в окно — минута `activeTo` уже вне активности.

### Интервал (`interval`)

- **interval** (integer, 1–1440): Как часто начинается новый цикл, в минутах.
  - Пример: `60` — каждый час, `30` — каждые полчаса, `1440` — раз в сутки

### Пауза (`duration`)

- **duration** (integer, 0–1440): Длительность фазы запуска в минутах.
  - `0` — второй блок действий не используется, startControls срабатывает один раз за интервал
  - Ограничение: `duration < interval`

### Дни недели (`scheduleDaysOfWeek`)

- **scheduleDaysOfWeek** (array): Дни, когда сценарий активен.
  - Возможные значения: `"monday"`, `"tuesday"`, `"wednesday"`, `"thursday"`, `"friday"`, `"saturday"`, `"sunday"`

### Действия запуска (`startControls`)

Массив действий, выполняемых в фазе запуска. Минимум 1 элемент.

Каждый элемент:
- **control** (string): Имя контрола в формате `"device/control"`
- **behaviorType** (string): Тип действия — `setEnable`, `setDisable`, `setValue`
- **actionValue** (number): Значение для `setValue`

### Действия остановки (`stopControls`)

Массив действий, выполняемых в фазе остановки. Игнорируется, если `duration = 0`.

Структура элементов аналогична `startControls`.

## Виртуальное устройство

Сценарий создаёт виртуальное устройство `wbsc_<idPrefix>` с контролами:

| Контрол | Тип | Описание |
|---|---|---|
| `rule_enabled` | switch | Включение/выключение сценария |
| `execute_now` | pushbutton | Ручной немедленный запуск `startControls` |
| `current_time` | text | Текущее системное время |
| `active_window` | text | Временное окно: «18:00 - 19:00» |
| `next_start` | text | Дата и время следующего запуска `startControls` |
| `next_stop` | text | Дата и время следующего отключения (только если `duration > 0`) |
| `state` | value | Состояние: «Активен» / «Ожидает» / «Отключен» |

## Особенности использования

1. **Действия выполняются каждую минуту:** Пока сценарий активен, в каждую минуту применяется нужная фаза. Если пользователь вручную изменил состояние контрола — в следующую минуту оно будет восстановлено сценарием.

2. **Формула activeTo:** `activeTo = время_последнего_старта + пауза + 1 мин`. Граница эксклюзивна: минута activeTo уже вне окна.

3. **Перезапуск wb-rules:** Автоматическая логика восстанавливается на следующей минуте. Очистка при выходе из окна и таймер ручного запуска не переживают перезапуск.

4. **Ручной запуск и авто-расписание независимы:** `execute_now` не сдвигает автоматическое расписание.

5. **Нет неидемпотентных типов:** `toggle`, `increaseValueBy`, `decreaseValueBy` недоступны — их многократный вызов даёт нежелательный накопительный эффект.

## Использование модуля из кода

```js
/**
 * @file: init-periodic-ventilation.js
 */

// Step 1: import module
var CustomTypeSc =
  require('periodic-timer.mod').PeriodicTimerScenario;

function main() {
  var scenarioName = 'Periodic ventilation';

  // Step 2: create instance
  var scenario = new CustomTypeSc();

  // Step 3: configure
  var cfg = {
    idPrefix: 'periodic_vent',  // optional
    activeFrom: '07:00',
    activeTo: '23:06',          // last start 23:00 + duration 5 + 1 = 23:06
    interval: 60,
    duration: 5,
    scheduleDaysOfWeek: [
      'monday', 'tuesday', 'wednesday',
      'thursday', 'friday', 'saturday', 'sunday'
    ],
    startControls: [
      {
        control: 'ventilation/fan',
        behaviorType: 'setEnable'
      }
    ],
    stopControls: [
      {
        control: 'ventilation/fan',
        behaviorType: 'setDisable'
      }
    ]
  };

  // Step 4: init
  try {
    var isInitSuccess = scenario.init(scenarioName, cfg);
    if (!isInitSuccess) {
      log.error(
        'Init aborted for scenario: "{}"',
        scenarioName
      );
      return;
    }
    log.debug('Initialized: "{}"', scenarioName);
  } catch (error) {
    log.error(
      'Exception: "{}" for scenario: "{}"',
      error.message || error,
      scenarioName
    );
  }
}

main();
```
