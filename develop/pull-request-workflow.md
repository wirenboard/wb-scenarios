# Работа с Pull Request

## Именование веток

- Новый функционал: `feat/INT-xxx-description`
- Исправление: `fix/INT-xxx-description`

Примеры:
- `feat/INT-871-periodic-timer`
- `fix/INT-878-default-scenario-name`

## Именование коммитов и PR

Формат: `<тип>: <описание> <номер задачи>`

- `feat: add periodic timer scenario INT-871`
- `fix: update astronomical timer offset INT-882`
- `fix: add enum for default scenario name INT-878`

Типы: `feat` — новый функционал, `fix` — исправление.

## Что приложить к PR

- Описание изменений
- Скриншоты веб-интерфейса (конфигуратор сценария) — если затронута схема
- Скриншоты страницы устройств — если затронуты виртуальные девайсы

## PR при создании нового сценария

Для нового сценария используется каскадный процесс PR, соответствующий
[этапам разработки](development-process.md):

1. Создаём ветку `feat/INT-xxx-scenario-name`
2. **Этап 1 — хардкод логики:** реализуем модуль сценария → создаём PR → проходим ревью
3. **Этап 2 — JSON-схема:** создаём ветку от PR, добавляем схему → новый PR → ревью → вливаем в первый PR
4. **Этап 3 — объединение:** создаём ветку от PR, подключаем init-модуль → новый PR → ревью → вливаем
5. Вливаем изначальный PR в main

## PR для обычных правок

- Создаём ветку `feat/` или `fix/`
- Вносим изменения
- Создаём PR с описанием и скриншотами (если UI затронут)
- Проходим ревью и вливаем
